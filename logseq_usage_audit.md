## 2. 实际记录习惯摘要

### 2.1 Daily Log 使用方式

每个 Daily Log 遵循一个高度一致的固定模板：

```
## Daily Note
    - ## 今日目标
        - {{embed ((63834265-4f84-4c7c-bd14-2b497928dd91))}}
    - ## 时间线 [[Daily Log]]
    - ## 上午
    - ## 下午
    - ## 晚上
    - ## 今日总结
        - ### 成绩<总结>
        - ### 自省<反思>
        - ### 有趣的事<趣事>
        - ### 感恩的事<感恩>
        - ### 碎碎念<随笔>
```

关键观察：
- **上午** 是填充最充分的时段，包含 TODOs、工作叙述、想法
- **下午** 和 **晚上** 经常为空
- Daily Log 是一种 **工作日记 + 任务管理 + 想法收集箱** 三位一体的混合体
- 记录语言为简体中文，风格自由，不使用过多标签分类
- 同一个 `{{embed ((63834265-...))}}` Daily Goals 模板块被嵌入到了数百个 journal 中，贯穿整个三年半的使用历史

### 2.2 TODO / DOING / DONE 使用方式

**TODO**:
- 格式: `- TODO <描述>`，位于任意缩进层级
- 2022-2023 年常用 `[#A]`/`[#B]`/`[#C]` 优先级标记 + `#执行清单`/`#愿望清单`/`#项目清单` 等 GTD 标签
- 2024-2026 年简化为纯 `TODO` 格式，不再使用优先级标记或 GTD 标签
- TODO 下方经常有子块，包含：具体的执行步骤、资源链接、备注、想法等

**DOING**:
- 在日常使用中非常罕见。最近 30 天 journals 中 DOING 数量为 0
- 项目页中偶有 DOING（共 7 个），且通常带有 `:LOGBOOK:` 时间追踪和详细的过程记录子块
- 这表明用户更倾向于在 TODO 状态下积累过程记录，而不是显式切换状态

**DONE**:
- 常附带 `:LOGBOOK:` 块记录 CLOCK 时间追踪（如 `CLOCK: [2026-04-24 Fri 13:52:43]--[2026-04-24 Fri 14:40:12] => 00:47:29`）
- 部分 DONE 块下方有完成内容的具体记录

### 2.3 项目页使用方式

- 大部分项目页以 `项目-` 前缀命名（如 `项目-Pravega.md`、`项目-海丝.md`）
- 但也存在用 `任务-`、`学习-`、`课程-` 等前缀命名的页面同样具有项目属性
- **高置信项目页**: 67 个（同时满足 `PARA:: [[PARA/Project]]` + 含项目结构标记）
- **中等置信项目页**: 30 个（满足其中一项）
- 项目页使用标准页属性: `PARA:: [[PARA/Project]]`、`Areas::`、`state::`、`priority::`、`start::`、`end::`
- 项目页内部使用固定结构标记: `[具体目标]`、`[具体事务]`、`[资源列表]`、`[头脑风暴]`、`[反思]`
- 部分项目额外使用: `[价值层]`、`[目标层]`、`[里程碑]`

### 2.4 想法条目使用方式

- **格式**: `**[想法]** <描述>` 为主，偶有 `[想法] <描述>` 和 `[随想]`
- **出现位置**:
  - 项目页的 `[头脑风暴]` 区域（最集中）
  - Daily Log 中作为独立条目
  - TODO/DOING 块的子块中（代表执行过程中的灵感）
- 大多数想法为单行捕获，少数有子块延展

### 2.5 块引用使用方式

- 两种形式: `((uuid))` 行内引用 和 `{{embed ((uuid))}}` 嵌入
- 全图谱约 1,434 处块引用
- 主要用途:
  - Daily Goals 模板的跨 journal 复用
  - 项目页中的内容复用和跨页关联
  - 学习笔记中的概念引用

### 2.6 过程记录使用方式

- 过程记录主要以 **子块** 形式存在于 TODO/DOING 下方
- 常见子块类型: `[注]`（备注）、`[想法]`（灵感）、`[问题]`（待解决问题）、`[部署]`（操作步骤）、代码块、URL 链接、嵌套的细化 TODO
- 日志式叙述（如 "现在我想维护一些台账...然后现在我的导师给我的任务是..."）也常见于 TODO 下方

### 3.5 是否可靠

- **TODO/DONE 识别高度可靠**: Logseq 使用严格的 `- TODO ...` / `- DONE ...` 格式，正则匹配准确率高
- **DOING 识别可靠但数量极少**: 格式同样严格，但用户日常使用中很少显式设置 DOING 状态
- **主要难点**:
  1. 区分"原始块"和"引用块"（`- ((uuid))` 形式的行只是引用，不是新任务）
  2. 嵌入块 `{{embed ((uuid))}}` 可能出现重复计数
  3. 代码块中包含 `TODO` 文本的情况会被误判（如 2026_05_05 日志中的分析提示）

### 4.4 项目页内部结构观察

**标准模板**（以 `项目-Pravega.md` 为典型）:

```
- TODO [#A] <项目名> #项目清单
  type:: [[项目]]
    - [具体目标]: <待完成的目标>
    - [具体事务]: <一些明确要做的内容>
        - TODO / DONE / DOING <具体任务>   ← 任务嵌入在事务区
          type:: [[任务]] / [[行动]]
          collapsed:: true
            - <详细子块: 操作步骤、代码、问题、视角等>
    - [资源列表]: <事务处理过程中你可以用到的资源>
    - [头脑风暴]: <事务执行过程中你的想法>
        - **[想法]** <具体想法>
        - **[随想]** <具体随想>
    - [反思]: <事务处理完成后回过头来你的反思>
```

**扩展模板**（以 `项目-海丝.md` 为典型）:

```
- [具体目标]:
    - [价值层] <OKR O: 为什么值得做>
    - [目标层] <OKR KR: 可验证里程碑>
    - [里程碑] <最小可执行单元>
- [具体事务]:
    - <阶段划分>
        - [阶段] / [子阶段]
            - [任务] / [子任务] / [待澄清]
```

**关键观察**:
- 标准模板（`[具体目标]` / `[具体事务]` / `[资源列表]` / `[头脑风暴]` / `[反思]`）在大多数项目页中都有体现
- 只有复杂项目才使用 `[价值层]` / `[目标层]` / `[里程碑]` 的扩展 OKR 分层
- `[具体事务]` 区域是真正承载 TODO/DOING/DONE 的地方
- 项目页中的 TODO 有独立的 `type:: [[任务]]` 或 `type:: [[行动]]` 元数据
- `collapsed:: true` 被广泛用于折叠详细内容

### 4.5 是否能稳定识别项目下的事务

**可以，但需注意**:
- 项目页中的 TODO/DOING/DONE 可被明确识别为"属于该项目的事务"
- 但同一个 TODO 如果同时出现在项目页和 journal 引用中，需要去重
- 部分事务位于 `[具体事务]` 区域内但没有显式 TODO 标记，可能被漏掉

  - `基于我的笔记做一款工作流或者历史事件方便查找的工具` 这个想法明显与当前正在做的 CLI 工具直接相关

#### 样本 1: Journal TODO + 深度子块（AI视频生成）

**来源**: `journals/2026_04_23.md`

**原始块**:
```

```

## 7. 给后续 CLI / Agent 接口的启示

### 7.1 后续系统最小应该暴露哪些对象

| 对象 | 包含内容 | 优先级 |
|------|----------|--------|
| **Task（事务）** | TODO/DOING/DONE 状态 + 标题 + 关联的子块记录 + 所属页面 | P0 |
| **Project（项目）** | 项目页元数据 + 下属事务列表 + 结构标记内容 + 想法/资源/反思 | P0 |
| **Idea（想法）** | 想法内容 + 出现位置 + 所属项目/事务（如有） | P1 |
| **JournalDay（日志日）** | 某一天的 TODO/想法/DONE 集合 | P1 |
| **BlockRef（块引用）** | UUID → 被引用位置的映射 | P2 |

### 7.2 哪些信息应该来自 Logseq 原始文件

- **块内容和层级关系**: 从 Markdown 缩进解析（4空格=1层级）
- **页属性**: 从文件头部的 `key:: value` 格式解析
- **任务状态**: 从 `- TODO/DOING/DONE` 前缀解析
- **时间追踪**: 从 `:LOGBOOK:` 块中的 `CLOCK:` 行解析
- **想法标记**: 从 `**[想法]**` / `[想法]` 标记解析
- **项目结构**: 从 `[具体目标]` / `[具体事务]` 等标记定位
- **块 UUID**: 从 `id::` 属性或 `((uuid))` 引用中提取

### 7.3 哪些信息可以作为索引缓存

- **全图谱 TODO/DOING/DONE 索引**: UUID → 状态 + 位置 + 子块数
- **项目页索引**: 项目名 → 文件路径 + 元数据 + 任务统计
- **Block reference 反向索引**: UUID → 所有引用位置
- **想法索引**: 按日期/项目分组的想法条目
- **Journal 活跃度摘要**: 每日 TODO/想法/DONE 计数

索引可以按日增量更新（只扫描新增/修改的 journal），项目页的变更频率远低于 journal。

### 7.4 哪些内容适合提供给 Agent

- **某个事务的完整上下文树**: 原始块 + 所有子块（过程记录、注记、想法）
- **项目页的当前状态摘要**: TODO 数量、最近活跃时间、优先级
- **被引用的块内容**: 当 Agent 看到 `((uuid))` 时，需能解析出原始块内容
- **最近 n 天的 Daily Log 摘要**: 让 Agent 了解近期工作流
- **按项目/优先级过滤的 TODO 列表**: Agent 可基于此帮助用户规划

**重要原则**: 不要一次提供全量数据。通过索引过滤后再按需加载具体内容。Agent 的上下文窗口有限，需要精准提供相关内容。

### 7.5 哪些操作不应该自动化

| 禁止操作 | 原因 |
|----------|------|
| 自动将想法转为 TODO | 想法需要人的判断和确认 |
| 自动修改 Logseq 原始文件 | 任何写入都必须用户明确授权 |
| 自动归类或重组项目结构 | 用户的项目模板是个人化的约定 |
| 自动创建/删除 TODO | 只在用户明确要求时操作 |
| 自动假设"无主块"的归属 | 可能属于某个上级事务的上下文 |
| 将结构性嵌入 `{{embed}}` 当作语义任务 | 模板复用 ≠ 任务关联 |

## 8. 风险与不确定性

### 8.1 可能误判的识别规则

| 规则 | 误判风险 | 缓解方式 |
|------|----------|----------|
| `- TODO ...` 正则匹配 | 代码块中的 TODO 文本被误判 | 需要感知 Markdown 代码块边界 |
| `PARA::` 属性识别项目 | 非项目页也可能用此属性 | 需结合命名前缀 + 结构标记交叉验证 |
| `[想法]` 标记识别 | 引用块中的想法被重复计数 | 区分原始块和引用块 |
| `- ((uuid))` 被当作任务 | 纯引用行不是新任务 | 需要显式检测纯 UUID 引用格式 |
| 缩进层级判定父子关系 | 不同主题下的缩进可能不可比 | 依赖 Logseq 一致的 4 空格缩进 |

### 8.2 不够稳定的结构

1. **项目内部结构的粒度不统一**: 有些项目用 `[价值层]/[目标层]/[里程碑]`，有些不用；有些用 `[阶段]/[子阶段]/[任务]`，有些不用。需要支持可配置的结构标记
2. **想法格式的历史变体**: 当前主流是 `**[想法]**`，但历史上出现过 `[想法]`、`#随想`、`#想法` 等变体
3. **DOING 状态使用率极低**: 用户习惯在实际工作中保持 TODO 状态，在下方直接写过程记录，而不是显式转为 DOING。这意味着"进行中"的事务需要通过其他信号（如最近的 journal 引用、下方有最近更新的子块）来推断
4. **Journal 与项目页的任务可能重叠**: 同一个任务可能先在 journal 中创建，后移动到项目页，导致重复计数

### 8.3 需要用户后续规范的内容

1. **DOING 状态的使用**: 如果希望追踪"进行中"的事务，可能需要更频繁地切换状态；或者，接受"有最近活动子块的 TODO = 进行中"作为隐式 DOING 的替代信号
2. **想法条目标记统一**: 建议统一为 `**[想法]**` 格式以方便稳定识别
3. **项目页命名约定**: `项目-` 前缀已形成惯例，但 `任务-`/`学习-`/`课程-` 等前缀的项目页是否应统一为 `项目-` 值得考虑
4. **敏感信息标记**: 部分工作日志中包含 token、密码、IP 地址等敏感信息。如果后续提供给 Agent，需要一个标记或过滤机制
5. **块引用的语义区分**: `{{embed ((uuid))}}` 用于模板复用（如 Daily Goals），`((uuid))` 用于内容引用——这两种语义不同，不应同等对待

### 8.4 不适合自动处理的场景

1. **模糊状态的任务描述**: 没有明确的 TODO/DOING/DONE 标记，但实际是任务的自由文本
2. **个人反思和决策类内容**: `[反思]`、`[头脑风暴]` 中的内容高度依赖上下文理解
3. **跨图谱引用**: 如果未来有多个 Logseq graph，跨图谱的引用无法自动解析
4. **图片和附件中的信息**: assets 中的 PDF 注释、截图中的文字无法通过纯文本解析获取
5. **经过多次编辑的块**: Logseq 的块可能在多次编辑后 UUID 不变但内容语义变化，历史追溯有难度

## 9. 结论

### 1. 能否较可靠地识别 TODO / DOING / DONE？

✅ **可以。** Logseq 严格使用 `- TODO/DOING/DONE` 格式，正则匹配准确率很高。本轮扫描识别到 901 个 TODO、7 个 DOING、402 个 DONE。主要需注意：过滤纯引用块 `((uuid))`、感知代码块边界、区分原始块和嵌入块。

### 2. 能否较可靠地识别项目页？

✅ **可以。** 通过 `PARA:: [[PARA/Project]]` + 项目结构标记双重验证，可识别 67 个高置信项目页。用户的实际项目概念比 `项目-` 前缀更广，需要接受 `学习-`/`任务-`/`课程-` 等前缀的页面也可能是有效的项目页。

### 3. 能否识别想法条目？

✅ **可以。** `**[想法]**` 格式明确且一致，共识别 588 条 standalone 想法。需要注意区分"独立想法"和"TODO 执行过程中的想法"——两者语义不同，应分别处理。

### 4. 能否把项目/事务/想法和它们下面的记录关联起来？

✅ **可以。** 通过 Markdown 缩进层级可稳定建立父子关系，块引用可通过 UUID 索引追踪跨页关联，项目结构标记可定位事务区域。5 个真实样本验证了从简单 TODO 到复杂项目页的完整上下文恢复能力。

### 5. 后续要做 CLI 的话，最小可行方向是什么？

**第一阶段 — 只读索引器**:
- 扫描 graph 生成 TODO/项目/想法的结构化索引
- 构建 UUID → 块位置的反向索引（用于解析块引用）
- 支持按日期/项目/状态过滤查询

**第二阶段 — 上下文查询**:
- 展示一个事务的完整上下文（原始块 + 子块树 + 所在项目页）
- 提供"今天的 TODO 及每个的上下文"视图
- 提供"某项目的当前状态"视图

**第三阶段 — Agent 集成**:
- 将索引数据作为 Agent 的上下文入口
- Agent 通过索引定位到相关块，再按需读取原始文件获取完整内容
- 核心原则保持不变: 只读优先，写入需用户授权；索引优先，按需加载；单图谱优先，后续扩展
