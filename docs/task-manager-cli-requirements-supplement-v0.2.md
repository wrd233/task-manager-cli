# task-manager-cli 个人行动系统需求设计补充稿

> 版本：v0.2  
> 定位：需求层 / 产品语义层 / 系统理念层  
> 用途：作为 `task-manager-cli-action-flow-system-design` 的深度补充，用于进一步明确系统边界、核心对象、流转语义、Logseq 写回策略、Clarify / Review / Proposal 机制、项目语义树、成果沉淀和人机协作方式。  
>
> 本文不进入代码架构、数据库结构、函数设计或实现细节。它关注“系统应该如何被理解、使用和约束”。

---

## 0. 本版设计的核心结论

经过进一步澄清，本系统的定位可以更明确地表述为：

> `task-manager-cli` 是一个以 CLI 为主要入口、以 Logseq 为自然记录现场、以行动条目为第一公民、以 Clarify / Review / Proposal 为流转机制、以项目语义树和成果沉淀为组织方式的个人行动系统。

它不是传统意义上的任务管理器，也不是简单的 Logseq TODO 抽取工具，而是一个长期个人行动系统：

```text
自然捕获
→ 行动条目抽象
→ Clarify 澄清
→ Proposal 建议
→ 用户审核
→ Logseq / 内部语义层同步
→ 项目语义树组织
→ 执行记录积累
→ 成果标注
→ 项目复盘与经验沉淀
```

其中：

- **CLI** 是用户高频操作入口；
- **CLI 同时给 Agent 提供操作空间**；
- **远端 API** 是日常 Clarify / Review 中的辅助处理能力；
- **Agent** 更多用于较重的分析、规划、诊断和项目级推理；
- **Logseq** 不只是数据源，而是可被同步、可被手动编辑、可承载项目语义树和成果标注的自然工作现场；
- **task-manager-cli 内部语义层** 用来维护比 Logseq 更结构化的行动状态、Proposal、Review、关系和历史；
- **用户确认** 是所有结构性变更的最终门槛。

本版设计相比上一版最重要的变化是：

1. Logseq 不再只是只读来源；经过用户确认后，部分语义可以写回。
2. GTD 状态既可以存在于内部语义层，也可以通过设计良好的 Logseq 标记表达。
3. 项目语义树倾向于与 Logseq 项目页结构同步，而不只是内部虚拟视图。
4. Annotation 更接近用户笔记中的 `[注]`，不应被 AI 大量污染。
5. AI 生成的批注应与用户批注区分为 `[AI注]`。
6. 成果沉淀不需要拆成过多抽象对象，优先统一为 `[成果]` 和 `[无成果]`。
7. Reference 不应简单视为 Action Item，而应更接近资源 / 支撑材料。
8. 日常 CLI 操作应该尽量短，面向人的命令可以考虑别名或更轻入口，不必每次显式输入复杂前缀。
9. 远端 API 与 Agent 是两类能力：日常 Clarify 可调用远端 API，但用户并不希望日常每个操作都由 Agent 接管。

## 1. 系统定位：个人行动系统，而非单纯 task manager

### 1.1 第一性定义

本系统要解决的问题不是“如何列出所有 TODO”，而是：

> 如何让一个人长期积累的想法、任务、项目、资源、过程记录和成果，能够在自然记录中产生，在需要行动时被澄清，在项目推进中被组织，在完成后被沉淀。

因此，系统关注的是完整行动生命周期，而不是单一任务状态。

```text
输入一条模糊内容
→ 判断它是什么
→ 决定放在哪里
→ 推进它
→ 记录过程
→ 完成后标注成果
→ 复盘成经验
```

这就是个人行动系统的核心。

### 1.2 CLI 的角色

CLI 是高频操作入口。

它承担两类使用者：

1. **用户本人**
   - 快速查看；
   - 快速输入；
   - 进入 Clarify；
   - 审核 Proposal；
   - 标注成果；
   - 查看项目；
   - 做 Review。

2. **Agent / 自动化工具**
   - 读取结构化上下文；
   - 发起诊断；
   - 创建 Proposal；
   - 查询对象；
   - 辅助项目 Review；
   - 读取上下文包进行计划分析。

因此，CLI 不只是给 Agent 的接口，也不只是给人的终端工具，而是系统的共同操作层。

### 1.3 设计文档层级

后续设计文档建议拆为三类：

1. **产品理念 / 需求设计文档**
   - 说明为什么需要系统、对象是什么、流转如何发生、边界在哪里。
   - 本文属于这一类。

2. **工程架构设计文档**
   - 说明模块、数据层、适配器、服务、命令、测试、扩展点。

3. **下一轮实施规格文档**
   - 明确下一步要做哪些功能，不做哪些功能，如何验收，并给 Codex / Claude Code 提示词。

这三类文档不应混在一起，否则会同时太抽象又太难执行。

## 2. 原始记录层与语义层：Logseq 和 task-manager-cli 的关系

### 2.1 Logseq 不是“只读数据源”

早期设计中更强调“不写回 Logseq”。这一原则对防止污染很重要，但现在需要更精确：

> Logseq 是自然记录现场，也是用户可以直接编辑的主工作空间；task-manager-cli 可以在用户确认后将部分语义写回 Logseq。

Logseq 中应继续承载：

- Daily Log；
- TODO / DOING / DONE；
- 项目页面；
- 项目语义树；
- `[想法]`；
- `[待澄清]`；
- `[注]`；
- `[AI注]`；
- `[成果]`；
- `[无成果]`；
- `#inbox` 等轻标签；
- 过程记录；
- 资源链接；
- 项目复盘。

### 2.2 task-manager-cli 内部语义层

内部语义层承担 Logseq 难以自然表达或不适合直接写入的内容：

- Proposal；
- Review Session；
- Proposal 历史；
- 回滚记录；
- 远端 API 处理状态；
- 用户审核状态；
- 关系置信度；
- 批量 Review 进度；
- 文件系统扫描候选；
- Agent 上下文缓存；
- 复杂的对象关系和派生关系。

### 2.3 同步原则

Logseq 和内部语义层应尽量一致，但不是简单“一字段双写”。

建议区分：

```text
Logseq 原始状态：来自 TODO / DOING / DONE / WAITING / 行首标注 / 标签
内部语义状态：系统澄清后的语义状态、Review 结果、Proposal 应用结果
```

同步原则：

1. Logseq 被用户直接修改后，内部应在同步时跟随更新。
2. 内部语义层的变更如果需要体现到 Logseq，必须经用户确认。
3. 用户确认后，可以写回 Logseq。
4. 写回必须遵守可读、可控、不乱的标记规范。
5. 内部不应默默覆盖用户手写内容。
6. 冲突时，应保留原文，并提示用户处理。

### 2.4 写回 Logseq 的可接受内容

可写回内容包括：

- 用户确认的新任务；
- 用户确认的 GTD 状态；
- 用户写的 `[注]`；
- 用户确认后的 `[AI注]`；
- 用户确认后的 `[成果]` / `[无成果]`；
- 用户确认的项目语义树节点；
- 用户确认的小任务；
- 用户确认的项目复盘摘要。

不应自动写回：

- 未确认 Proposal；
- Agent 草稿判断；
- 批量分类临时结果；
- 文件自动关联候选；
- 远端 API 中间结果；
- 高风险结构调整。

## 3. 行动条目：第一公民

### 3.1 行动条目的定义

行动条目是系统中的第一公民。

它代表一个可被流转、处理、组织、执行或沉淀的行动单位。

它可以来自：

- Logseq TODO；
- Daily Log 中的待处理记录；
- CLI inbox；
- Flomo；
- 项目页；
- `[待澄清]`；
- `[想法]` 演化出的事务；
- Agent 提议；
- 用户手动创建。

行动条目不是所有信息的总称。它强调“与行动有关”。

### 3.2 Reference 不是行动条目

Reference 更适合理解为资源，而不是 Action Item。

区别是：

| 类型 | 核心问题 | 是否需要推进 |
|---|---|---|
| Action Item | 我要不要处理它？下一步是什么？ | 是 |
| Reference | 它能支撑哪些行动或判断？ | 否 |

Reference 可以关联行动条目、小任务、项目节点或项目，但本身不进入行动流转。

因此：

```text
Reference = Resource / 支撑材料
Action Item = 可流转行动单位
```

### 3.3 行动条目的多重语义

行动条目可以有多重语义。不同语义意味着它应该被如何处理，而不是单一标签。

例如：

- 一个条目可以是 Idea，同时是 Someday；
- 一个条目可以是 Next Action，同时 Waiting；
- 一个条目可以是 Done，同时缺少成果标注；
- 一个条目可以是 Inbox，同时疑似属于某项目；
- 一个条目可以是 Project Candidate，同时还未正式建项目。

因此，不应强迫所有条目只有一个语义。

推荐理解：

```text
行动条目有主处理状态，也可以有辅助语义标签。
```

其中主处理状态用于决定它进入哪个视图和 Review 队列；辅助语义用于描述它的特点。

## 4. Idea：既可以转化为事务，也可以作为设计对象持续发展

### 4.1 Idea 不是“低级 Task”

Idea 不是一个未完成的 Task，也不是所有模糊内容的垃圾桶。

Idea 有两类重要形态：

1. **贴近事务的 Idea**
   - 随着发展，可以转化成一个 Task 或小任务。

2. **贴近设计的 Idea**
   - 它本身就是一种思考、设计、原则、方案或结构；
   - 它不应被直接消耗掉；
   - 它可能长期保留，并在发展过程中孕育出多个行动条目。

### 4.2 Idea 转化与孕育的区别

#### 转化

当 Idea 很贴近一个具体事务时，可以转化为行动条目。

例如：

```text
Idea：返点公司可以做风险评分表
→ Task：整理返点公司风险评分表字段
```

这种情况下，可以保留来源关系：

```text
Task derived_from Idea
```

#### 孕育

当 Idea 是一种设计、原则或方案时，不建议直接“转化掉”。

例如：

```text
Idea：项目语义树可以和 Logseq 项目页结构同步
```

它可能孕育出：

- 设计文档任务；
- adapter 调整任务；
- Logseq 标记规范任务；
- Review 流程设计任务。

但这些任务不一定挂在 Idea 下面，而应在这些任务上下文中说明：

```text
此任务源自 / 服务于某个设计型 Idea。
```

### 4.3 Idea 的归属

Idea 可以属于：

- Project；
- Project Node；
- 小任务；
- Someday；
- Daily Log；
- Inbox；
- 独立设计空间。

当 Idea 属于 Project 或 Node 时，它不是简单待办，而是一种设计补充、思路储备或潜在方向。

### 4.4 Idea 的保留原则

设计型 Idea 应尽量保留，不要因为生成了 Task 就删除或覆盖。

系统应支持：

- Idea 持续演化；
- Idea 派生行动；
- Idea 关联项目；
- Idea 进入 Someday；
- Idea 被标记为成熟 / 待发展 / 暂缓 / 已吸收。

## 5. 小任务：最小 PARA/Project，但独立于 Project

### 5.1 小任务的定义

小任务是比行动条目更大的轻量组织单元。

它可以理解为：

> 最小、最精简的 PARA/Project。

但它不等同于正式 Project。

小任务的特点：

- 可以独立存在；
- 可以出现在 Daily Log；
- 可以归属于 Project；
- 可以归属于 Project Node；
- 可以纳管多个行动条目；
- 可以纳管资源；
- 可以有成果；
- 可以作为项目树中的 Workstream；
- 可以由 Agent 建议创建。

### 5.2 小任务和 Project 的区别

| 维度 | 小任务 | Project |
|---|---|---|
| 时间跨度 | 短到中 | 中到长 |
| 结构复杂度 | 轻量 | 可复杂 |
| 是否需要 Logseq 项目页 | 不一定 | Logseq 中的 Project 应对应项目页 |
| 是否需要 PARA::Project | 不需要 | 当前 Logseq Project 需要 |
| 是否可独立存在 | 可以 | 可以 |
| 成果 | 通常有 | 通常有多个 |
| 复盘价值 | 局部 | 较强 |
| 例子 | 整理打车攻略初稿 | 韩国旅行准备 |

### 5.3 小任务的嵌套

小任务可以嵌套，但不鼓励无限嵌套。

需求层建议：

- 允许嵌套；
- 保持浅层；
- 可以设定推荐上限，例如 3 层；
- 深层嵌套时，系统应提示是否应该提升为 Project 或重构为项目节点。

### 5.4 小任务的成果要求

小任务不一定一开始就有清晰成果，但完成时应检查成果标注。

如果小任务完成了，但下面没有：

```text
**[成果]** ...
```

也没有：

```text
**[无成果]** ...
```

系统应在 Review 或 Clarify 中提醒。

## 6. Project：正式项目与 Logseq 项目页

### 6.1 Logseq 中的 Project

在 Logseq 中，Project 应对应项目页。

当前规范倾向：

```text
PARA:: [[PARA/Project]]
```

也就是说，Logseq 中的正式 Project 应具有明确页面和项目属性。

### 6.2 CLI 内部创建 Project

Project 可以由 CLI 内部创建，但如果它要成为 Logseq 中的正式项目，最终应能创建或绑定到一个 Logseq 项目页。

CLI 内部创建的项目可以先处于草稿状态：

```text
Project Candidate / Draft Project
```

用户确认后再写回 Logseq 项目页。

### 6.3 什么可以算 Project

只要满足以下特征，就可以作为 Project：

- 步骤层次多；
- 资源需求大；
- 有持续时间；
- 有多个行动条目或小任务；
- 可以作为工作中的项目被推进；
- 有复盘和成果沉淀价值。

例如：

- 学习计划；
- 旅行准备；
- 工作交付；
- 生活目标；
- 技术项目；
- 报告体系；
- 内容系统搭建。

## 7. GTD 语义流转与 Logseq 标记

### 7.1 GTD 是处理语义，不是标签崇拜

GTD 状态的意义在于决定条目如何被处理。

因此可以多重语义并存。

例如：

```text
一个条目可能是 idea + someday；
一个条目可能是 next action + waiting；
一个条目可能是 done + missing_result;
```

### 7.2 状态可以部分写回 Logseq

GTD 状态不必全部写回，但可以通过设计良好的方式写回。

可用表达包括：

1. **Logseq 原生状态**
   - TODO；
   - DOING；
   - DONE；
   - WAITING。

2. **行首标注**
   - `**[待澄清]**`；
   - `**[想法]**`；
   - `**[成果]**`；
   - `**[无成果]**`；
   - `**[注]**`；
   - `**[AI注]**`。

3. **行末标签**
   - `#inbox`；
   - `#someday`；
   - `#waiting`；
   - `#reference`。

### 7.3 写回原则

写回的主要目标是“不乱”。

因此：

- 高频自然状态可以写回；
- 明确对用户阅读有帮助的状态可以写回；
- 系统内部辅助状态不一定写回；
- AI 建议必须确认后才能写回；
- 批量写回应有预览；
- 写回格式要稳定、可读、可被 adapter 再次识别。

### 7.4 内外状态一致性

目标是尽量一致。

- Logseq 改了，内部同步时应更新；
- 内部改了，经过用户批准后可同步到 Logseq；
- 未批准的内部建议不能覆盖 Logseq。

## 8. Done 与成果标注

### 8.1 Done 不是结束

Done 表示行动完成，但不表示成果已沉淀。

系统应关注：

```text
DONE 之后是否有 **[成果]** 或 **[无成果]** 标注。
```

### 8.2 成果统一为 `[成果]`

相比拆分 Outcome / Artifact / Knowledge 等复杂概念，当前需求层更适合统一表达为：

```text
**[成果]** ...
```

如果没有成果，也显式标注：

```text
**[无成果]** ...
```

这有几个好处：

- 符合 Logseq 自然书写；
- 易于 Review；
- 易于 adapter 抽取；
- 不引入过多对象复杂性；
- 用户能直接理解；
- 方便 Agent 判断项目进展。

### 8.3 成果的内容可以多样

`[成果]` 下方可以描述：

- 文档；
- 表格；
- 代码；
- 文件路径；
- 报告；
- 链接；
- 经验；
- 复盘；
- 可复用方法；
- 记忆卡片草稿。

也就是说，系统内部可以从 `[成果]` 中进一步识别文件、经验和知识，但用户层面不需要先区分 Outcome / Artifact / Knowledge。

### 8.4 经验卡片的触发

Done 后可以提供生成经验卡片的建议，但不应自动生成。

应该由用户显式触发，例如：

```text
markcard
```

或者在某个 DONE 条目下主动调用相关命令。

原则：

> 经验卡片生成是用户主动触发的沉淀动作，不是每个 Done 的默认负担。

## 9. Annotation：用户高频批注与 AI 注释

### 9.1 Annotation 的重新定位

Annotation 不应主要理解为 Agent 批注系统。

它更接近用户笔记中的：

```text
**[注]** ...
```

也就是用户会高频使用的补充说明。

Annotation 可以存在于多个层次：

- 行动条目；
- 小任务；
- Project；
- Project Node；
- 成果；
- 文件关联；
- Review Session。

### 9.2 用户注与 AI 注

需要区分：

```text
**[注]** 用户批注
**[AI注]** AI 批注
```

AI 或远端 API 可以提供批注，但不应大量写入，避免冲散原文信息。

AI 注的原则：

- 尽量短；
- 必须用户确认；
- 不替代用户原文；
- 不大量堆积；
- 可折叠或可过滤；
- 适合作为辅助解释，而非行动事实。

### 9.3 Annotation 与 Proposal 的区别

Annotation 是批注，不是结构变更。

Proposal 是可应用的结构化变更。

例如：

```text
[注] 这个任务其实依赖客服回复
```

是 Annotation。

```text
将 #123 状态改为 WAITING
```

是 Proposal。

两者可以关联，但不能混用。

---

## 10. Proposal：可审核、可回滚的结构化变更

### 10.1 Proposal 的定义

Proposal 是系统或 AI 对内部语义层 / Logseq 表达提出的结构化变更建议。

它可以建议：

- 改 GTD 状态；
- 改标题；
- 创建小任务；
- 归入项目；
- 挂到项目节点；
- 写回 `[AI注]`；
- 标记 `[成果]` / `[无成果]`；
- 创建项目树节点；
- 移动项目树节点；
- 关联文件；
- 归档条目；
- 合并条目；
- 拆分条目。

### 10.2 风险分级

Proposal 应按风险分级。

#### 低风险

- 建议分类；
- 建议状态；
- 建议 relation；
- 添加内部注释；
- 标记待澄清。

#### 中风险

- 创建小任务；
- 创建项目节点；
- 关联文件；
- 写回 `[AI注]`；
- 标记 `[成果]` / `[无成果]`。

#### 高风险

- 删除；
- 合并；
- 大规模项目树重组；
- 批量写回 Logseq；
- 改写原文标题；
- 移动大量节点。

高风险必须显式确认，不应批量自动接受。

### 10.3 可回滚

Proposal 必须可追溯，并需要支持回滚。

原因：

- AI 会误判；
- 用户当时可能确认错；
- 项目理解会变化；
- 批量 Review 可能产生系统性错误。

设计原则：

```text
每次 Proposal 应用都应留下足够信息，以便撤销或修正。
```

---

## 11. Clarify：主动进入的语义澄清流程

### 11.1 Clarify 入口

Clarify 应支持多个入口：

- Inbox Clarify；
- Today Clarify；
- Project Clarify；
- Idea Clarify；
- Historical Task Clarify；
- Manual Selection Clarify。

### 11.2 Clarify 交互模式

采用混合模式：

1. 系统 / 远端 API 给出初步建议；
2. 对不确定或高影响条目提出问题；
3. 用户逐条回答；
4. API 后台处理；
5. 生成 Proposal；
6. 汇总成建议表；
7. 用户审核；
8. 应用确认的 Proposal。

### 11.3 问题来源

Clarify 的问题来源：

```text
固定基础问题 + 远端 API 补充问题
```

固定问题保证一致性，远端 API 问题用于复杂条目。

### 11.4 Clarify 输出

Clarify 不直接修改对象，而是输出 Proposal。

用户审核后才能应用。

---

## 12. Review Session：Clarify 的组织单位

### 12.1 Review Session 的意义

Review Session 是一次可暂停、可恢复、可审核的批量处理过程。

它记录：

- 本次 Review 的范围；
- 处理了哪些条目；
- 用户回答了什么；
- 远端 API 生成了什么；
- 生成了哪些 Proposal；
- 用户接受或拒绝了哪些；
- 应用了哪些变更；
- 哪些条目跳过了；
- 哪些需要后续继续。

### 12.2 Review 类型

Review 可以包括：

- Inbox Review；
- Today Review；
- Task Review；
- Idea Review；
- Project Review；
- Tree Review；
- Done Review；
- Weekly Review。

### 12.3 Review 的交互形态

允许三种形态：

1. 普通 CLI；
2. 交互式 CLI prompt；
3. TUI Review 界面。

其中，Clarify 和批量审核天然适合 TUI，但不应强制只能 TUI。

---

## 13. 项目语义树：与 Logseq 项目页结构同步的一等对象

### 13.1 项目树的定位

项目语义树是一等公民，但不是每个项目必须拥有复杂树。

它应与 Logseq 项目页结构同步。

也就是说：

- 用户可以在 Logseq 中直接编辑项目树；
- CLI 可以读取项目树；
- Agent 可以提出项目树变更 Proposal；
- 用户确认后可以写回 Logseq；
- 内部可以维护项目树的结构化理解。

### 13.2 Logseq 中的树节点表达

项目树节点倾向使用行首标识。

例如：

```text
**[目标]** ...
**[里程碑]** ...
**[工作流]** ...
**[小任务]** ...
**[具体事务]** ...
**[资源]** ...
**[成果]** ...
```

这些节点不是强制所有项目都要有。

一个项目可以非常简单：

```text
Project
- TODO action 1
- TODO action 2
```

也可以复杂：

```text
Project
- **[目标]**
  - ...
- **[里程碑]**
  - ...
- **[工作流]**
  - ...
- **[小任务]**
    - TODO ...
```

### 13.3 核心节点类型

优先核心节点：

1. Workstream / 工作流 / 小任务组；
2. 小任务；
3. Action Item；
4. Resource；
5. 成果。

Objective、Key Result、Milestone 可以存在，但不是必需。

换句话说：

> 项目树不应强迫每个项目都 OKR 化。

### 13.4 树为主，引用补充

项目结构以树为主，避免烧脑。

多重从属尽量通过 Logseq 引用实现，而不是把项目语义层设计成复杂图。

原则：

```text
主归属尽量唯一；
补充关联可以通过引用和 related relation 表达。
```

### 13.5 AI 优化项目树

AI 可以提出：

- 新增节点；
- 删除节点；
- 合并节点；
- 调整节点顺序；
- 改写节点描述；
- 将行动条目挂入节点；
- 将想法挂入节点；
- 发现节点缺少行动；
- 发现节点缺少成果。

全部作为 Proposal，不自动应用。

---

## 14. 项目纳管

### 14.1 纳管的定义

项目纳管是将散落的行动条目、想法、资源、成果纳入某个 Project 或 Project Node 的语义范围。

它可以表现为：

- 内部 relation；
- Logseq 引用；
- 写回项目页；
- 项目树挂载；
- Review 确认记录。

### 14.2 纳管不一定移动原文

如果条目在 Daily Log 中产生，可以不移动原文。

可以通过：

- 引用；
- relation；
- 项目页汇总；
- 内部挂载；
- Logseq block ref；

实现纳管。

### 14.3 纳管优先级

优先通过明确线索纳管：

1. 条目位于项目页；
2. 条目引用项目；
3. 条目在项目节点下；
4. 用户明确指定；
5. AI 推断。

AI 推断必须经用户确认。

---

## 15. 文件系统与成果关联

### 15.1 文件系统作为辅助证据源

文件系统可作为项目成果证据源，但初期只做 metadata 扫描。

扫描内容：

- 文件名；
- 路径；
- 类型；
- 修改时间；
- 大小；
- 最近新增 / 修改。

不默认读取文件内容。

### 15.2 文件关联必须确认

文件与行动条目、小任务、项目或成果的关联都应先成为 Proposal。

用户确认后才建立正式关联。

### 15.3 文件与 `[成果]`

如果某个 `[成果]` 提到了文件路径，系统可以识别为成果证据。

例如：

```text
**[成果]** 形成了 `/path/to/report.docx`
```

文件系统扫描也可以反向提出：

```text
这个新文件可能对应某条 DONE 的成果。
```

但仍需用户确认。

---

## 16. 远端 API 与 Agent 的分工

### 16.1 远端 API

远端 API 是日常操作中的辅助处理能力。

它用于：

- Clarify；
- 分类建议；
- 描述细化；
- 生成 Proposal；
- Review 辅助；
- 批量语义判断。

远端 API 应可配置。

它可以是 DeepSeek、OpenAI、其他模型服务，或未来自定义服务。

### 16.2 Agent

Agent 是更重的分析者。

它用于：

- 项目诊断；
- 复杂计划；
- 多上下文推理；
- 代码修改；
- 大规模系统设计；
- 深度 Review；
- 使用 `tm agent` 上下文包。

日常操作中不一定每次都使用 Agent。

### 16.3 API 调用模式

Clarify 的用户交互与 API 处理应解耦。

推荐模式：

1. 用户回答问题；
2. 系统收集上下文；
3. 后台提交远端 API；
4. API 返回 Proposal；
5. 用户审核。

支持：

- 查看处理状态；
- 失败重试；
- 批量处理；
- 暂停恢复。

## 17. 隐私与安全

### 17.1 开发阶段的现实取舍

当前作为个人工具和开发计划，可以先以方便为主，不必过早把脱敏机制做得过重。

但需求文档仍需保留长期安全边界：

- 远端 API 可能接触个人信息；
- 项目中可能有工作敏感信息；
- 文件路径和内容可能敏感；
- token / cookie / 密码不能长期裸奔。

### 17.2 建议原则

当前阶段：

```text
以本地个人使用、快速迭代为主。
```

长期阶段：

```text
支持 payload 预览、可配置脱敏、敏感信息提示、文件内容默认不发送。
```

这样既不阻碍早期开发，也不牺牲未来安全扩展。

## 18. 人类入口与命令体验

### 18.1 日常命令应尽量短

面向人的操作不应每次都需要复杂前缀。

可以考虑：

- `tm` 作为完整命令；
- 简短别名；
- 交互式 shell；
- TUI 首页；
- 默认进入 view / review 模式；
- Agent 命令保留明确前缀。

例如：

```text
today
inbox
clarify
review
project
done
```

这些可以是 `tm` 下的交互式命令，也可以是 shell alias。

### 18.2 视图分层

建议保留四层视图：

```text
view    给用户快速看
review  给用户交互式澄清
agent   给 Agent 深度分析
report  给开发者调试
```

面向用户的输出要短、清楚、少字段。

面向 Agent 的输出可以深。

面向开发者的 report 可以全量。

## 19. 成功标准

系统设计成功的标志不是“功能多”，而是：

1. 用户可以低摩擦捕获。
2. 历史 TODO / Idea 的混乱能逐步降低。
3. Clarify 像做题一样自然，不增加太多负担。
4. Proposal 让 AI 建议可审核、可回滚。
5. Logseq 中的状态和内部语义尽量一致。
6. 项目树既能表达结构，又不强迫所有项目复杂化。
7. 小任务能承接多步骤事务，避免所有东西都变成 Project。
8. Done 后能提示成果缺失。
9. `[成果]` 能成为项目进展的重要证据。
10. 文件系统能补充项目真实产出，但不制造噪声。
11. Annotation 不被 AI 污染，用户注释仍保持主导。
12. Agent 和远端 API 各司其职。
13. 用户始终有最终确认权。

## 20. 对上一版设计的补充关系

本文不是推翻上一版 `task-manager-cli-action-flow-system-design`，而是对其进行精确化。

主要补充点包括：

1. 明确系统定位为个人行动系统。
2. 明确 Action Item 是统一抽象，但 Reference 不应视为 Action Item。
3. 明确 Idea 可转化，也可作为设计型对象长期保留。
4. 明确小任务独立于 Project，可独立存在，也可挂载项目。
5. 明确 Logseq Project 应对应项目页和 `PARA::Project`。
6. 明确 GTD 状态可以部分写回 Logseq。
7. 明确 Done 后用 `[成果]` / `[无成果]` 标注。
8. 明确 Annotation 更接近用户 `[注]`，AI 注应为 `[AI注]`。
9. 明确 Proposal 是结构变更，必须可回滚。
10. 明确项目语义树应与 Logseq 项目页结构同步。
11. 明确项目树以 Workstream / 小任务 / Action Item 为核心。
12. 明确文件系统初期只做 metadata 扫描。
13. 明确远端 API 与 Agent 是不同角色。
14. 明确日常 CLI 操作要尽可能短。

## 21. 总结

本系统可以被最终理解为：

> 一个围绕 Logseq 自然记录建立的个人行动系统：它以行动条目为第一公民，以小任务和项目语义树组织复杂行动，以 Clarify / Review / Proposal 处理语义流转，以 `[成果]` 承接完成后的沉淀，以 CLI 提供高频操作入口，并通过远端 API 与 Agent 提供不同层级的智能辅助。

它的关键不是让 AI 接管行动，而是让自然记录逐步变得可行动、可组织、可复盘、可沉淀。

最简洁的系统口号可以是：

```text
先自然记录，再逐步澄清；
先形成行动，再沉淀成果；
AI 负责建议，用户负责确认；
Logseq 保持现场，CLI 提供流转。
```
