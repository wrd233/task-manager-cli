# task-manager-cli 行动流转系统需求设计稿

> 版本：v0.1  
> 定位：需求层 / 系统语义层设计  
> 背景来源：基于现有 `task-manager-cli` 设计、Logseq 使用习惯、后续能力构想，以及本轮关于 Clarify、行动条目、小任务、GTD 状态、项目语义树的补充想法整理。  
>
> 本文不进入代码层细节，不限定数据库表、类名、函数签名或具体实现框架。它关注的是：系统应该解决什么问题、有哪些核心对象、对象如何流转、人与 Agent 如何协作、哪些信息应该留在 Logseq，哪些应该进入 `task-manager-cli` 内部语义层。

---

## 0. 总体判断

`task-manager-cli` 的目标不应停留在“任务管理器”或“Logseq TODO 索引器”。

从第一性原理看，它真正要承担的是：

> 将分散在 Logseq、Flomo、CLI 输入、本地文件系统中的行动线索，抽象为可被持续澄清、组织、推进、复盘和沉淀的行动对象，并为用户和 Agent 提供稳定的上下文接口。

这意味着系统关注的不是单个 TODO，而是一个长期循环：

```text
自然捕获
→ Clarify 澄清
→ 语义归位
→ 项目纳管
→ 执行与记录
→ 成果确认
→ 经验沉淀
→ 项目复盘
→ 再进入下一轮行动
```

在这个循环中：

- **Logseq** 是自然记录和原始工作现场；
- **task-manager-cli** 是行动对象、关系、语义状态、Review 记录和成果索引的内部系统；
- **Agent** 是语义加工、分类建议、项目诊断、结构重塑和成果提炼的助手；
- **用户** 是最终确认者、选择者和执行者。

核心边界是：

> 系统可以建议，但不应擅自替用户确认；系统可以建立内部语义层，但不应强制重写 Logseq 原文；系统可以辅助项目结构重塑，但不能把 AI 的理解直接当成事实。

## 1. 第一性原理：这个系统到底解决什么问题

### 1.1 不是“我有哪些任务”，而是“我的行动世界现在是什么状态”

用户在真实工作和生活中产生的信息不是标准任务列表，而是混合形态：

- 今天随手写下的 TODO；
- 项目页里的目标和事务；
- 某个 TODO 下方的过程记录；
- `[想法]` 和 `[随想]`；
- Flomo 中的碎片想法；
- 已经完成但没有沉淀成果的事项；
- 本地文件夹中的文档、表格、脚本、报告；
- Agent 对项目的批注和建议；
- 被反复拉回 Daily Log 的历史任务。

这些信息共同构成一个“行动世界”。

系统的第一性任务是：

> 从这些分散的自然记录中恢复当前行动世界，并使其可以被澄清、组织、推进和复盘。

### 1.2 不是“把所有东西结构化”，而是“在合适阶段逐步增加结构”

输入时，人最需要的是低摩擦。

如果一开始就要求用户填写项目、优先级、GTD 状态、Key Result、成果形式、上下文、截止日期，系统会变成负担。

因此设计原则是：

```text
Capture 阶段低结构；
Clarify 阶段增结构；
Review 阶段确认结构；
执行与沉淀阶段积累结构。
```

也就是说：

- 刚捕获时，只要不丢；
- 进入 Clarify 时，再询问必要问题；
- Agent 给出建议后，再由用户确认；
- 确认后，才进入系统内部的稳定语义层。

### 1.3 不是“AI 自动管理我”，而是“AI 辅助我做语义判断”

Agent 的价值不在于替用户做最终决定，而在于：

- 把粗糙描述变清晰；
- 判断条目可能属于什么语义状态；
- 推荐归属项目或项目节点；
- 发现重复、阻塞、待澄清；
- 建议项目结构如何调整；
- 在完成后帮助提炼成果和经验。

但这些都应该先成为建议，而不是事实。

所以系统必须区分：

```text
原始记录
系统抽取事实
Agent 判断
Agent 建议变更
用户确认结果
```

## 2. 系统核心对象：行动条目是第一公民

### 2.1 行动条目 Action Item

本系统中最核心的对象不是 Task、Project 或 Idea，而是更底层的：

> 行动条目。

行动条目代表一个可以被捕获、澄清、组织、推进、关联、完成或沉淀的基本单位。

一个行动条目可以是：

- 还未澄清的 inbox 条目；
- 一个想法；
- 一个明确行动；
- 一个等待事项；
- 一个未来也许做的事情；
- 一个资料型记录；
- 一个已完成但待沉淀成果的事项。

行动条目的关键特征：

1. **有来源**
   - 来自 Logseq、Flomo、CLI 输入、本地文件系统或 Agent 建议。

2. **有原始文本**
   - 不因 AI 重写而丢失原始表达。

3. **有当前位置**
   - 可以回到 Logseq 块、Flomo 记录、本地文件或 CLI 内部记录。

4. **有语义状态**
   - 例如 inbox、next action、idea、waiting、someday、reference、done 等。

5. **可以被组织**
   - 归入小任务、项目、项目节点、成果、知识卡片等结构中。

6. **可以被 Review**
   - Agent 可以提出分类、改写、归属、拆分、合并、沉淀建议。

7. **可以保留历史**
   - 语义状态变化、用户确认、Agent 建议都应该留痕。

### 2.2 为什么行动条目是第一公民

用户补充了一个重要想法：

> 行动条目之上的各种逻辑概念，本质上都是对行动条目和资源的聚合。

这意味着系统不应过早把结构固定为“项目大于任务，任务大于子任务”的单一层级。

更合理的理解是：

```text
行动条目是原子行动单位；
小任务、项目、Objective、Key Result、阶段、成果等，都是围绕行动条目形成的组织视图。
```

这样的好处是：

- 一个行动条目可以被重新归类；
- 项目结构可以重塑；
- 同一个条目可以同时与项目、资源、成果、经验有关；
- 不需要因为项目树变化而移动原始 Logseq 块。

## 3. 小任务：介于行动条目和项目之间的最小组织单元

### 3.1 小任务的必要性

有些事情单个行动条目无法表达，因为它需要分多步完成。

例如：

```text
完成韩国旅行打车攻略
```

它可能包含：

- 查询 Kakao T 使用方法；
- 对比出租车和机场接送；
- 整理付款方式；
- 写一版给女朋友看的说明；
- 检查常见坑。

这些行动条目共同服务于一个更小的目标。

这时可以形成一个“小任务”。

### 3.2 小任务与项目的关系

小任务可以理解为：

> 最小粒度的 Project，或者轻量 Project。

用户给出的比喻非常准确：

```text
小任务像进程；
行动条目像线程；
进程纳管了一组线程和一些资源。
```

在系统语义上，小任务具有以下特征：

- 它比行动条目大；
- 它比正式项目小；
- 它有一个相对明确的产出；
- 它包含若干行动条目和资源；
- 它可以归属于某个项目；
- 它也可以暂时独立存在，后续再纳入项目。

### 3.3 小任务的适用场景

小任务适合用于：

1. **短周期多步骤事务**
   - 例如“整理某个攻略”“完成某个报表”“处理某次演练材料”。

2. **项目下的一个工作包**
   - 例如“搭建 APM 报告 SQL 章节结构”。

3. **尚未上升为项目，但明显超过单个行动的事务**
   - 例如“搭一版 task-manager-cli GTD Review 流程”。

4. **项目语义树中的叶子级聚合节点**
   - 例如某个 Key Result 下的一个具体工作包。

### 3.4 小任务与 GTD 的关系

在 GTD 中，如果一个结果需要多个行动，就应该视为 Project。

但在用户系统中，“Project”已经承载了更大的长期项目语义，因此可以引入“小任务”作为轻量项目。

语义层次可以是：

```text
行动条目 < 小任务 < 项目 < 项目组合 / Area
```

但这种层次不应强制所有对象都完整具备。它只是系统理解复杂度的工具。

## 4. Clarify 阶段：主动进入的澄清流程

### 4.1 Clarify 的定位

Clarify 不是后台自动分类，而是用户主动进入的阶段。

它解决的问题是：

> 系统中有大量粗糙、混乱、历史遗留、语义不明确的行动条目，需要逐步澄清它们到底是什么。

Clarify 既可以针对单条，也可以针对一批。

例如：

- 今天新增的条目；
- 最近 7 天的未澄清条目；
- 某个项目下的无主条目；
- 所有 suspicious idea；
- 一批历史 TODO；
- 某个 inbox 队列。

### 4.2 Clarify 的交互形式：像做题一样

用户提出的交互非常关键：

> 命令行像给我做题一样，一个个展示行动条目当前的样子，然后给出问题让我回答；我填好回答后自动跳到下一个问题；这个问题交给远端 API 处理；全部回答完并处理完后，显示一张当前建议状态表，我可以审核和修改。

这是一个非常适合 CLI/TUI 的模式。

流程可以抽象为：

```text
选择 Review 范围
→ 展示第 1 个行动条目
→ 系统/Agent 提出澄清问题
→ 用户回答
→ 跳转下一个条目
→ 后台提交远端 API
→ 收集所有建议
→ 生成建议状态表
→ 用户批量审核
→ 用户确认后应用到内部语义层
```

### 4.3 Clarify 中应该问什么问题

不同条目需要的问题不同，但可以设计一套基础问题库。

#### 对所有条目都可能问

- 这条是否仍然有价值？
- 这是一个行动、想法、资料，还是未来可能？
- 它是否属于某个已有项目？
- 它是否需要进一步拆分？
- 它是否已经完成或过期？
- 它是否需要等待别人？
- 它是否只需要归档为参考资料？

#### 对疑似 Task 的条目

- 下一步动作是否足够明确？
- 是否能在一次行动中完成？
- 是否需要转成小任务？
- 是否有明确成果？
- 是否有截止或外部承诺？
- 是否需要关联资源？

#### 对疑似 Idea 的条目

- 它只是灵感，还是可以转成任务？
- 它是否可以并入已有项目？
- 它是否值得保留？
- 是否需要发展成项目？
- 是否只是资料或反思？

#### 对疑似 Project / 小任务的条目

- 它期望达成什么结果？
- 最小可交付成果是什么？
- 第一批行动条目是什么？
- 是否已有相关资源？
- 是否有已有项目可以承接它？

### 4.4 用户回答与远端 API 的分工

用户回答的是“个人判断、意图、偏好和上下文”。

远端 API 处理的是：

- 基于回答细化描述；
- 推荐 GTD 状态；
- 推荐归属项目或小任务；
- 推荐是否拆分；
- 推荐是否转为小任务；
- 推荐是否作为 reference / someday；
- 生成结构化 Proposal。

用户不需要一次性想清楚全部结构，只需要回答系统提出的问题。

### 4.5 Clarify 输出：建议状态表

Clarify 结束后，不应立即应用所有修改。

应该生成一张建议状态表，供用户审核。

表格可以包含：

| 条目 | 当前状态 | Agent 建议 | 建议原因 | 用户操作 |
|---|---|---|---|---|
| #123 韩国打车攻略 | inbox | 转为小任务，归入韩国旅行 | 需要多步完成 | 接受 / 修改 / 拒绝 |
| #124 返点风险评分 | idea | 归入韩国旅行头脑风暴 | 是方法想法 | 接受 / 修改 / 拒绝 |
| #125 等客服回复 | todo | waiting | 依赖外部回复 | 接受 / 修改 / 拒绝 |

用户可以：

- 接受；
- 拒绝；
- 修改后接受；
- 暂缓；
- 批量接受低风险建议；
- 保存 Review 进度后退出。

### 4.6 Clarify 的核心原则

1. **用户主动进入**
   - 不在后台擅自改变语义。

2. **问题驱动**
   - 系统通过问题帮助用户澄清，而不是直接给结论。

3. **Agent 提建议**
   - Agent 负责加工和结构化。

4. **用户审核**
   - 所有结构性修改必须经用户确认。

5. **只写内部语义层**
   - 默认不修改 Logseq 原文。

## 5. GTD 状态是否需要写入 Logseq

### 5.1 问题本质

用户提出：

> GTD 的几个状态是否需要在 Logseq 中用特定方式标注？

这个问题本质上是在问：

> GTD 状态应该属于原始记录层，还是属于 task-manager-cli 的语义层？

### 5.2 推荐答案：默认不要强制写入 Logseq

原则上，GTD 状态应该优先维护在 `task-manager-cli` 内部，而不是强制写入 Logseq。

原因：

1. **避免破坏自然记录**
   - Logseq 当前的优势是低摩擦记录。

2. **避免双向同步复杂化**
   - 如果 Logseq 和 CLI 都维护状态，容易冲突。

3. **GTD 状态是后验澄清结果**
   - 它通常不是输入当下就知道的。

4. **Agent 的判断可能变化**
   - 今天是 idea，明天可能成为 project；不应频繁改原文。

5. **内部语义层更适合批量 Review**
   - 可以接受、拒绝、撤销、重做。

### 5.3 但 Logseq 可以保留轻量提示

虽然不建议强制写入，但可以允许用户自愿使用少量标记。

例如：

```text
#等待
#maybe
#reference
```

或子块：

```text
- [等待] 等对方回复
- [资源] xxx
- [想法] xxx
```

这些标记可以作为 adapter 的辅助信号，但不应成为唯一依据。

### 5.4 推荐分层

#### Logseq 层

保留自然表达：

- TODO / DOING / DONE；
- `[想法]`；
- `[资源]`；
- `[问题]`；
- `[等待]`；
- 项目页结构；
- Daily Log。

#### task-manager-cli 层

维护正式语义：

- gtd_state；
- clarified_status；
- belongs_to relation；
- project node relation；
- proposal history；
- review result；
- annotation。

#### Agent 层

进行判断：

- 这是什么；
- 该归到哪里；
- 是否需要拆分；
- 是否应该 waiting；
- 是否可以丢弃；
- 是否要沉淀成果。

### 5.5 最终原则

> Logseq 中可以有提示性标记，但 GTD 状态的主事实源应在 task-manager-cli 内部语义层。

## 6. Proposal：AI 建议变更的核心载体

### 6.1 为什么需要 Proposal

系统中会出现大量 AI 建议：

- 把某条 inbox 转成 task；
- 把某个 task 转成小任务；
- 把某个 idea 归入项目；
- 把某条标记为 waiting；
- 把某条归档为 reference；
- 把某组任务挂到某个 Key Result；
- 建议新增项目节点；
- 建议删除过时节点；
- 建议关联某个文件作为成果。

这些建议都不是事实。

它们需要成为一种可审核对象：

> Proposal。

### 6.2 Proposal 与 Annotation 的区别

#### Annotation

是评论、判断、说明。

例如：

```text
这个项目可能卡在“想法多但缺少明确下一步”。
```

#### Proposal

是可应用的结构化变更建议。

例如：

```text
将 #123 从 inbox 改为 next_action，并关联到 Project #45。
```

因此：

```text
Annotation = 解释性信息
Proposal = 可执行变更
```

### 6.3 Proposal 的状态

Proposal 应该至少有以下状态：

- suggested；
- accepted；
- rejected；
- edited；
- applied；
- expired；
- superseded。

其中：

- accepted 代表用户接受建议；
- applied 代表已经写入内部语义层；
- rejected 代表用户明确拒绝；
- edited 代表用户修改后采纳；
- superseded 代表被新的建议替代。

### 6.4 Proposal 的来源

Proposal 可以来自：

- Clarify 流程；
- 项目 Review；
- Inbox Review；
- Done Review；
- 项目语义树优化；
- 文件系统扫描；
- Agent 主动诊断。

### 6.5 Proposal 的应用边界

默认只应用到 `task-manager-cli` 内部：

- 改变内部 GTD 状态；
- 建立 relation；
- 挂到项目节点；
- 记录成果；
- 记录 annotation；
- 建立文件关联。

不默认修改 Logseq 原始文件。

如果未来允许写回 Logseq，也必须是独立的显式流程。

## 7. Review Session：批量澄清和批量确认的工作单元

### 7.1 为什么需要 Review Session

Clarify、批量归类、项目纳管、成果沉淀，都需要“可暂停、可恢复、可审核”的流程。

如果只是一次性调用 Agent，很难保证：

- 用户能逐条确认；
- 进度不会丢；
- 结果可追溯；
- 中途可以暂停；
- 后续可以继续；
- 错误建议可以撤销。

因此需要 Review Session。

### 7.2 Review Session 的类型

系统可以支持多种 Review：

1. **Inbox Review**
   - 澄清未处理条目。

2. **Today Review**
   - 处理今天新增和变动。

3. **Task Review**
   - 清理历史 TODO。

4. **Idea Review**
   - 整理想法。

5. **Project Review**
   - 诊断项目和纳管散落条目。

6. **Done Review**
   - 对完成事项做成果沉淀。

7. **Weekly Review**
   - 周度整理项目、等待项、长期未完成项。

8. **Tree Review**
   - 优化项目语义树。

### 7.3 Review Session 的过程

典型过程：

```text
选择范围
→ 生成候选条目
→ 逐条展示
→ 用户回答问题
→ Agent 生成 Proposal
→ 汇总建议表
→ 用户审核
→ 应用已确认 Proposal
→ 记录 Review 结果
```

### 7.4 Review Session 的人机交互原则

1. **像做题一样逐条推进**
   - 每次只处理一个条目，降低认知负担。

2. **可批量接受低风险建议**
   - 例如明显的 reference、明显的 waiting。

3. **支持跳过**
   - 对不想处理的条目保留待处理状态。

4. **支持编辑建议**
   - 用户不必只能接受或拒绝，可以改后采纳。

5. **支持暂停和恢复**
   - 大批量历史条目不可能一次处理完。

6. **保留 Review 记录**
   - 之后可以知道某条状态为什么这么定。

---

## 8. 项目语义树：从静态页面到动态结构

### 8.1 项目语义复杂化是不可避免的

用户提出：

> 项目下面的树结构不同类型的具体语义是什么？它们之间是否有从属关系？当我们希望对项目进行优化时，AI 对这些树结构的新增、修改、删除建议是有价值的，但也可能带来项目语义复杂化。

这是系统设计中的一个核心张力。

项目需要结构，否则无法管理复杂性；但结构太多，又会成为负担。

### 8.2 项目树的第一性理解

项目树不是为了好看，也不是为了模拟组织架构。

项目树的作用是：

> 把项目的目标、阶段、任务、资源、成果和经验组织成一个可理解、可诊断、可推进的结构。

它应该服务于三个问题：

1. **为什么做**
   - 价值、目标、动机。

2. **做到什么算成**
   - Key Result、里程碑、成果。

3. **怎么推进**
   - 小任务、行动条目、资源、文件、下一步。

### 8.3 项目树节点的语义类型

项目树可以包含不同类型节点，但需要明确它们的含义。

#### 1. Objective：目标 / 方向

表示项目想达成的高层意图。

特征：

- 偏“为什么”和“方向”；
- 不一定直接可执行；
- 可以包含多个 Key Result；
- 适合承载价值层和目标层。

示例：

```text
让韩国旅行准备进入可执行状态，并恢复对承诺的掌控感。
```

#### 2. Key Result：可验证结果

表示 Objective 下可判断是否完成的结果。

特征：

- 应该尽量可验证；
- 可以对应一组小任务或成果；
- 应该回答“做到什么算完成”。

示例：

```text
形成一份可读的韩国打车攻略初稿。
```

#### 3. Milestone：里程碑

表示项目推进中的阶段性节点。

特征：

- 偏时间或阶段；
- 可以跨多个 Key Result；
- 用于阶段推进和回顾。

示例：

```text
出发前一周完成所有旅行准备策略。
```

#### 4. Workstream：工作流 / 工作包

表示一类持续推进的工作线。

特征：

- 适合承载多个小任务；
- 不一定有单一完成点；
- 适用于长期项目中的并行方向。

示例：

```text
返点公司筛选
```

#### 5. Mini Project / 小任务组

表示需要多个行动才能完成的轻量目标。

特征：

- 比单个行动大；
- 比正式项目小；
- 通常有明确产出；
- 可以纳管多个行动条目和资源。

示例：

```text
整理韩国打车攻略初稿
```

#### 6. Action Item：行动条目

表示可以推进的基本行动单元。

特征：

- 系统第一公民；
- 可以归属到任意树节点；
- 可以有 GTD 状态；
- 可以有过程记录和成果。

示例：

```text
查询 Kakao T 是否需要韩国手机号。
```

#### 7. Resource：资源

表示用于支持行动或判断的资料。

特征：

- 不一定要执行；
- 支持项目理解和行动；
- 可以是链接、文件、笔记、截图。

#### 8. Outcome / Artifact：成果

表示已经产出的结果。

特征：

- 可被引用；
- 可对应文件；
- 可用于复盘；
- 可支撑项目完成度判断。

#### 9. Knowledge / Lesson：经验

表示从项目或行动中抽象出的可复用知识。

特征：

- 脱离单次任务仍有价值；
- 可以转成 Anki Card；
- 可以进入方法论库。

### 8.4 节点之间的关系

常见从属关系：

```text
Project
└── Objective
    └── Key Result
        └── Mini Project / Workstream
            └── Action Item
                └── Outcome / Knowledge
```

但这不是唯一结构。

更灵活的关系应该包括：

- supports：某资源支持某目标；
- contributes_to：某行动贡献到某 Key Result；
- produces：某行动产出某成果；
- derived_from：某任务由某想法转化而来；
- blocks：某条目阻塞某条目；
- depends_on：某条目依赖某条目；
- evidence_for：某文件是某成果证据。

因此，项目树不应被理解为单一父子树，而应被理解为：

> 以树为主视图、以图关系补充的项目语义结构。

### 8.5 AI 对项目树的优化建议

AI 可以建议：

- 新增 Objective；
- 拆分 Key Result；
- 合并重复节点；
- 删除过时节点；
- 将任务移动到更合适节点；
- 将无主想法挂到某节点；
- 发现某节点缺少下一步行动；
- 发现某 Key Result 缺少成果；
- 发现某项目目标和现有任务不匹配。

但所有这些建议都应成为 Proposal。

用户确认后，才改变内部项目语义树。

### 8.6 项目语义复杂化的控制原则

为了避免项目树变成负担，需要几个约束：

1. **项目树是可选视图**
   - 不是所有项目都必须建树。

2. **行动条目仍是第一公民**
   - 树只是组织方式，不是行动本身。

3. **节点类型不应过多**
   - 初期可以只支持 Objective、Key Result、小任务、行动条目、资源、成果。

4. **AI 建议不自动应用**
   - 避免结构被频繁重塑。

5. **复杂项目才需要复杂树**
   - 小项目可以只有小任务和行动条目。

6. **原始 Logseq 层级不强制同步**
   - 语义树可以存在于内部系统中。

## 9. 项目纳管：把散落条目归入项目而不移动原文

### 9.1 纳管的第一性定义

项目纳管不是把 Logseq 块移动到项目页。

项目纳管是：

> 在内部语义层建立“这个行动条目、想法、资源、成果与这个项目有关”的关系。

因此，一个散落在 Daily Log 的 TODO，只要建立 relation，就已经被项目纳管。

### 9.2 纳管来源

系统可以发现候选纳管关系：

- 条目中出现项目 wiki link；
- 条目出现在项目页；
- 条目出现在项目页的子块；
- 条目标题与项目名称相似；
- 条目包含项目相关关键词；
- Agent 基于上下文推断；
- 用户手动指定；
- 文件系统成果与项目目录相关。

### 9.3 纳管 Review

项目级 Review 可以做：

```text
给定某个项目
→ 找出可能相关但未纳管的行动条目、想法、资源、文件
→ Agent 给出关联建议
→ 用户确认
→ 建立内部 relation
```

### 9.4 纳管的价值

纳管后，项目诊断会更准确：

- 项目下有哪些未完成行动；
- 项目有哪些散落想法；
- 哪些成果文件已经存在；
- 哪些条目长期未处理；
- 哪些目标没有行动支撑；
- 哪些行动没有成果输出。

---

## 10. 成果沉淀：Done 之后发生什么

### 10.1 Done 不应该是终点

在当前很多任务系统中，Done 是结束。

但在这个系统里，Done 应该成为成果沉淀入口。

完成一个行动条目后，系统可以问：

- 这件事产出了什么？
- 是否有文件？
- 是否有文档？
- 是否有经验？
- 是否需要复盘？
- 是否可以转成知识卡片？
- 是否支撑某个 Key Result？
- 是否应该更新项目进展？

### 10.2 成果对象

成果可以是：

- 文档；
- 表格；
- 脚本；
- 配置；
- 报告；
- PPT；
- 截图；
- 代码提交；
- 经验总结；
- Anki Card；
- 项目复盘片段。

成果对象的作用是：

> 让系统知道“做完了什么”，而不仅是“打了一个勾”。

### 10.3 成果与文件系统

如果成果存在于本地文件系统，应允许关联到文件。

例如：

```text
Task #123 完成
→ 产出 haisi_pmis_plan.docx
→ 文件关联到 Task #123
→ Task #123 的 Outcome 指向该文件
```

文件不只是附件，而是项目进展的证据。

### 10.4 成果沉淀与经验复用

完成事项时，Agent 可以帮助提炼：

- 问题本质；
- 处理步骤；
- 判断依据；
- 可复用模板；
- 注意事项；
- Anki Card；
- 复盘段落。

这样系统会从任务管理走向经验积累。

---

## 11. 文件系统关联：把项目成果纳入上下文

### 11.1 为什么需要文件系统关联

很多项目的真实成果不在 Logseq 里，而在本地文件夹中。

例如：

- Word 报告；
- Excel 台账；
- Markdown 设计文档；
- Python 脚本；
- 截图；
- PPT；
- 配置文件；
- 代码仓库。

如果 Agent 只看 Logseq，就只能看到计划和记录，看不到真实产出。

因此系统需要支持：

> Project 与本地 root folder 的关联。

### 11.2 项目根目录

每个 Project 可以配置一个或多个 root folder。

系统可以扫描：

- 最近新增文件；
- 最近修改文件；
- 文件类型；
- 文件名；
- 文件路径；
- 目录结构；
- 可能关联的任务或成果。

### 11.3 文件关联的边界

文件扫描不应自动得出强结论。

它可以提出候选：

```text
这个文件可能是 Task #123 的成果。
```

然后用户确认。

### 11.4 文件系统对项目诊断的价值

项目诊断时，Agent 可以参考：

- 哪些成果文件已经存在；
- 哪些文件最近更新；
- 是否缺少预期交付物；
- 文件结构是否混乱；
- 是否需要拆分、归档、命名规范化；
- 某些任务是否已经有实际产出但未标记 Done。

---

## 12. 用户与 Agent 的协作边界

### 12.1 Agent 可以做什么

Agent 可以：

- 帮助澄清条目；
- 生成问题；
- 重写描述；
- 判断 GTD 状态；
- 推荐归属；
- 推荐拆分；
- 推荐小任务；
- 推荐项目树调整；
- 推荐成果沉淀；
- 生成批注；
- 生成 Proposal。

### 12.2 Agent 不应做什么

Agent 不应：

- 未经确认直接改变内部语义层；
- 未经确认写回 Logseq；
- 未经确认移动项目树节点；
- 自动删除或归档条目；
- 把建议当成用户原始记录；
- 把推断当成事实。

### 12.3 用户保留最终确认权

所有结构性变化都应经过用户确认：

- GTD 状态变化；
- 项目归属变化；
- 小任务创建；
- 项目树调整；
- 文件关联；
- 成果确认；
- 经验卡片生成。

## 13. 典型场景设计

### 13.1 场景一：捕获一个新事务

用户输入：

```text
韩国旅行打车攻略
```

系统行为：

1. 捕获为 inbox action item；
2. 记录来源和时间；
3. 可选进入 Clarify；
4. Agent 询问：
   - 这是旅行项目的一部分吗？
   - 期望成果是什么？
   - 是一个动作还是多步事务？
5. Agent 建议：
   - 转为小任务；
   - 归入韩国旅行项目；
   - 目标为“形成打车攻略初稿”；
   - 拆出 3 个行动条目。
6. 用户确认或修改；
7. 系统写入内部语义层。

### 13.2 场景二：批量澄清历史 TODO

用户启动：

```text
Review 最近 30 天未澄清条目
```

系统行为：

1. 列出一批候选；
2. 逐条展示；
3. 提出问题；
4. 用户回答；
5. Agent 生成建议；
6. 汇总建议表；
7. 用户批量接受或修改；
8. 系统应用确认后的语义变更。

### 13.3 场景三：项目级纳管

用户选择项目：

```text
韩国旅行
```

系统行为：

1. 找出散落在 Daily Log、Idea、Flomo 中的相关条目；
2. Agent 判断可能关联；
3. 给出纳管建议；
4. 用户确认；
5. 系统建立 relation；
6. 项目视图中出现这些条目。

### 13.4 场景四：项目树优化

用户进入项目树 Review：

1. 系统展示当前 Objective / Key Result / 小任务 / 行动条目；
2. Agent 检查结构问题；
3. 提出新增、合并、删除、移动建议；
4. 用户确认；
5. 系统更新内部项目语义树；
6. Logseq 原文不变。

### 13.5 场景五：完成任务后的成果沉淀

用户完成一个行动条目：

1. 系统检测 Done；
2. 进入 Done Review；
3. Agent 询问：
   - 产出了什么？
   - 是否有文件？
   - 是否有经验？
   - 是否可转成卡片？
4. 用户提供文件或说明；
5. 系统建立 Outcome；
6. 可选生成 Anki Card 草稿；
7. 成果进入项目复盘上下文。

## 15. 关键设计原则

### 15.1 行动条目第一

所有复杂结构都围绕行动条目组织，而不是反过来让行动条目被结构绑死。

### 15.2 Capture 不施压

输入阶段越简单越好，不强迫分类。

### 15.3 Clarify 增结构

语义结构应在 Review / Clarify 中逐步增加。

### 15.4 AI 建议必须可审核

AI 不能直接改系统事实。所有结构性建议都应成为 Proposal。

### 15.5 Logseq 保持自然现场

Logseq 原文不应被频繁重写。GTD 状态、项目语义树、Proposal、Review 结果优先存在于内部系统。

### 15.6 项目结构可以动态重塑

项目树是当前理解，不是永久真理。

### 15.7 成果比完成更重要

Done 之后应尽量沉淀 Outcome、Artifact 和 Knowledge。

### 15.8 文件是成果证据

项目文件夹是项目进展的重要证据源，应纳入上下文。

### 15.9 用户最终确认

无论 Agent 多强，最终归类、纳管、结构调整、成果确认都由用户决定。

## 16. 仍需进一步明确的问题

1. 行动条目、小任务、项目三者的边界如何在 UI 中表现？
2. 小任务是否需要单独的状态，还是继承项目状态？
3. GTD 状态是否支持多选？
   - 例如一个条目既是 idea，又是 someday？
4. Proposal 应该能否批量应用？
5. 批量应用是否需要置信度阈值？
6. Review Session 是否需要 TUI？
7. 项目语义树初期支持多少节点类型合适？
8. 项目树是否允许一个行动条目挂到多个节点？
9. 文件系统扫描是否只做 metadata，还是读取文件内容？
10. 文件关联是否需要用户确认？
11. 成果沉淀是否在每次 Done 时触发，还是只在 Review 中触发？
12. Anki Card 草稿如何与现有知识系统衔接？
13. 内部语义层是否需要版本历史和回滚？
14. Logseq 中的 GTD 轻标记是否要提供推荐规范？
15. 如何避免系统因过度 Review 而增加负担？

## 17. 总结

当前系统未来的方向可以概括为：

> 从“行动对象索引器”升级为“行动条目流转、项目语义组织与成果沉淀系统”。

其核心不是让 AI 替用户管理人生，而是建立一套可持续的人机协作机制：

```text
人自然捕获
→ 系统保留来源
→ Clarify 阶段逐步提问
→ Agent 生成建议
→ 用户审核确认
→ 内部语义层更新
→ 项目持续组织
→ 行动完成后沉淀成果
→ 文件和经验成为长期资产
```

最关键的几个概念是：

- 行动条目第一公民；
- 小任务作为轻量项目；
- GTD 状态作为内部语义层；
- Clarify 作为主动澄清阶段；
- Proposal 作为 AI 建议变更载体；
- Review Session 作为批量确认机制；
- 项目语义树作为动态组织结构；
- Outcome / Artifact / Knowledge 作为完成后的沉淀对象。

如果这些概念建立起来，`task-manager-cli` 就不只是一个 CLI 工具，而会成为一个围绕用户自然记录习惯构建的长期行动操作系统。
